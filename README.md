# Sprint Planner

Визуальный планировщик спринтов для команды: помогает разложить задачи по людям и фазам, увидеть план на таймлайне, учесть отпуска, командные выходные и тестовые активности, а также быстро найти перегрузы.

<img width="1683" height="667" alt="Sprint Planner timeline" src="https://github.com/user-attachments/assets/7ad4495b-8ea9-4f3e-b713-c9c402f46923" />

> Sprint Planner не заменяет Jira или другой трекер. Это локальный инструмент для планирования картины спринта; финальные задачи можно отдельно переносить в основной трекер.

---

## Возможности

- Планирование задач в рамках конкретного спринта.
- Разбивка каждой задачи на фазы, обычно `Dev / Review / Test`.
- Назначение фаз на участников с учетом ролей.
- Внешний `Review` без нагрузки на команду.
- Учет отпусков, командных выходных, регресса и смоука.
- Автоматический перенос фаз через выходные и недоступные дни.
- Подсветка конфликтов, когда дневная загрузка участника превышает 8 часов.
- Таймлайн с drag-and-drop, изменением длительности фаз и масштабом на 1 или 2 недели.
- Отчет по спринту с датами фаз, исполнителями и ссылками на задачи.
- Полностью локальное хранение в браузере через `localStorage`.

## С чего начать

### 1. Добавь участников

Открой **Участники** в правом верхнем углу, введи имя и роль участника.

Роли используются в интерфейсе планирования:

| Роль | Как влияет |
|------|------------|
| Разработчик / Developer / Dev | Может быть исполнителем обычных фаз |
| Тестировщик / QA / Tester / Test | Доступен для фазы `Test` |

Каждому участнику автоматически назначается цвет. Участников можно редактировать и удалять. При удалении участника он снимается с назначенных фаз задач.

### 2. Настрой спринт

Кликни по названию спринта в шапке, чтобы изменить дату начала, название или длительность. По умолчанию спринт длится 14 календарных дней.

Стрелки `‹` и `›` переключают соседние спринты. Задачи и события привязаны к дате старта спринта, поэтому при переключении показывается только план выбранного спринта.

Эпоха нумерации спринтов начинается с `2026-04-13` — это первый понедельник.

### 3. Выбери масштаб таймлайна

Переключатель **1 нед / 2 нед** в шапке меняет горизонтальный масштаб:

- **1 нед** — крупнее дни, удобно детально двигать блоки.
- **2 нед** — весь стандартный спринт виден компактнее.

Настройка сохраняется в браузере.

### 4. Добавь события

Нажми **Событие** в шапке, чтобы добавить блок занятости или недоступности.

| Тип | Для кого | Что делает |
|-----|----------|------------|
| Отпуск | Один участник | Делает участника недоступным; фазы этого участника переносятся дальше |
| Выходной | Вся команда | Делает всех недоступными; фазы задач не ставятся на эти дни |
| Регресс | Один участник | Занимает время и может создать конфликт |
| Смоук | Один участник | Занимает время и может создать конфликт |

Длительность можно вводить как `2д`, `4ч`, `1д 4ч`, `2d`, `4h` или простым числом часов. Рабочий день равен 8 часам.

События можно перетаскивать по таймлайну. Правый клик по событию открывает редактирование.

### 5. Создай задачу

Нажми **Задача** или кликни по пустой ячейке таймлайна. При клике по ячейке новая задача сразу получает выбранный день старта и подходящего первого исполнителя.

В форме задачи доступны:

- **Название** — например `PROJ-42 Авторизация`.
- **Ссылка** — URL на задачу в трекере; отображается в отчете.
- **Цель спринта** — задача помечается значком и поднимается в отдельную группу.
- **Дата начала** — только в пределах текущего спринта.
- **Фазы** — список работ с исполнителем и оценкой.

Если расчетная дата завершения выходит за границы спринта, форма подсвечивает предупреждение.

## Фазы задач

По умолчанию создаются фазы `Dev`, `Review` и `Test`, но их можно добавлять, удалять, менять местами и переименовывать выбором из списка.

| Фаза | Поведение |
|------|-----------|
| Dev | Обычная рабочая фаза, назначается на участника |
| Review | Внешняя фаза без исполнителя и без нагрузки на команду |
| Test | Назначается только на участников с тестовой ролью |

Оценка фазы поддерживает форматы `8h`, `2d`, `1d 4h` или простое число часов. В интерфейсе также встречаются русские обозначения `д` и `ч` для событий.

Фазы идут последовательно. Если фазу сдвинуть на таймлайне отдельно, для нее сохраняется ручной отступ `startAfterDays` от старта задачи.

Выходные, командные выходные и отпуск конкретного исполнителя пропускаются при расчете дат. Регресс и смоук не переносят расписание автоматически, но учитываются в загрузке и конфликтах.

## Таймлайн

Таймлайн — основная рабочая область. Строки — участники, столбцы — дни спринта.

- Участники отсортированы так, чтобы тестировщики были выше разработчиков.
- В левой части строки показаны роль, занятость по рабочим дням и мини-индикатор загрузки.
- Выходные подсвечиваются отдельно.
- Текущий день отмечается вертикальной линией.
- Конфликтные дни подсвечиваются красным.
- Для `Review` есть отдельная строка **Внешние ревьюеры**.

Действия на таймлайне:

| Действие | Результат |
|----------|-----------|
| Клик по пустой ячейке | Создать задачу на выбранного участника и день |
| Перетаскивание блока задачи | Сдвинуть задачу целиком |
| Перетаскивание `Test` или `Review` | Сдвинуть отдельную фазу внутри задачи |
| Перетаскивание события | Сдвинуть событие |
| Потянуть правый край фазы | Изменить длительность фазы |
| Клик по фазе | Выделить задачу и показать связи между фазами |
| Правый клик по фазе или событию | Открыть редактирование |
| Наведение на фазу | Показать тултип с деталями |

На длинных фазах отображаются часы по дням, чтобы было видно, как дробная длительность раскладывается по календарю.

## Конфликты и загрузка

Загрузка считается по людям и дням. Один рабочий день — 8 часов.

Конфликт возникает, когда суммарная дневная нагрузка участника больше 1 рабочего дня. В расчет входят:

- фазы задач с реальным исполнителем;
- отпуск;
- командный выходной;
- регресс;
- смоук.

В расчет конфликтов не входят:

- внешние `Review`-фазы;
- фазы без реального исполнителя.

При конфликте:

- строка участника и конкретный день подсвечиваются;
- фазы, участвующие в конфликте, получают красную подсветку;
- в шапке появляется счетчик конфликтных задач;
- в левой панели задача попадает в группу **С конфликтами**.

---

## Левая панель задач

Левая панель показывает только задачи текущего спринта.

Группы:

| Группа | Что попадает |
|--------|--------------|
| С конфликтами | Задачи, у которых есть конфликтные фазы |
| Цели спринта | Задачи, отмеченные как цель спринта |
| Запланированы | Обычные задачи, которые есть на сетке |
| Не на сетке | Задачи без рассчитанных блоков |

Доступны поиск по названию, открытие задачи кликом и удаление через кнопку `×` на карточке.

---

## Отчет по спринту

Кнопка **Отчет** переключает приложение из таймлайна в табличный отчет.

Отчет показывает:

- задачи текущего спринта;
- отметку **Цель спринта**;
- ссылку на задачу, если она заполнена;
- даты `Dev`, `Review` и `Test`;
- исполнителей для рабочих фаз;
- общий диапазон задачи и суммарную оценку.

Задачи в отчете сортируются так же по приоритету цели спринта: сначала цели, затем остальные.

---

## Хранение данных

Все данные хранятся в браузере в `localStorage`, ключ — `sprint-planner-v2`.

Сохраняются:

- настройки текущего спринта;
- участники;
- задачи и фазы;
- события;
- выбранный масштаб таймлайна.

Данные не отправляются на сервер. При очистке данных браузера план будет потерян.

## Разработка

Стек: React 19, TypeScript, Vite, Tailwind CSS v4. Бэкенда нет.

```bash
npm install
npm run dev
```

Локальный сервер Vite обычно доступен на `http://localhost:5173`.

Полезные команды:

```bash
npm run build    # production-сборка в dist/
npm run lint     # eslint
npm run preview  # предпросмотр production-сборки
```

## Структура проекта

```text
src/
  types.ts                    # интерфейсы данных
  store.ts                    # useAppStore(), localStorage и миграции
  conflicts.ts                # расписание, загрузка, конфликты
  App.tsx                     # корневой компонент, шапка, модалки, переключение видов
  components/
    TimelineGrid.tsx          # основная сетка таймлайна и drag-and-drop
    LeftPanel.tsx             # список задач слева
    TaskEditor.tsx            # создание и редактирование задач
    EventEditor.tsx           # создание и редактирование событий
    PeopleManager.tsx         # участники команды
    SprintSettings.tsx        # настройки спринта
    SprintReport.tsx          # отчет по спринту
    WorkloadBar.tsx           # визуализация загрузки
    Modal.tsx                 # обертка модального окна
```

Основные правила поддержки:

- новые типы добавлять в `src/types.ts`;
- новые расчеты и алгоритмы держать в `src/conflicts.ts`;
- все мутации состояния делать через `useAppStore()`;
- напрямую в `localStorage` вне `store.ts` не писать;
- при изменении структуры данных добавлять миграцию в `loadState()`;
- не добавлять серверный стейт — приложение намеренно локальное.
